# 架構設計引導 (/creative)

## 設計理念
這不是一個要填充的表格，而是架構思考的引導框架。根據專案特性自由調整深度和風格。

## 核心問題引導

### 1. 架構願景 (Architecture Vision)
**思考引導：**
- 這個系統要解決什麼核心問題？
- 5年後這個系統會是什麼樣子？
- 什麼是絕對不能妥協的架構原則？

**表達建議：**
- 可用故事敘述、隱喻或圖像
- 重點是傳達「為什麼這樣設計」

### 2. 系統分解 (System Decomposition)
**思考引導：**
- 系統的自然邊界在哪裡？
- 哪些部分變化頻率高？哪些相對穩定？
- 團隊結構如何影響系統劃分？

**表達自由度：**
- C4 Model (Context, Container, Component)
- Domain-Driven Design (Bounded Contexts)
- Service-Oriented Architecture
- 或任何適合的分解方式

### 3. 關鍵決策 (Key Decisions)
**思考引導：**
- 哪些技術選擇會長期影響系統？
- 有哪些 trade-offs？
- 什麼驅動了這些決策？

**記錄方式：**
- 可用 ADR 格式
- 或簡單的決策矩陣
- 或敘述性說明

### 4. 品質屬性 (Quality Attributes)
**思考引導：**
- 性能、安全、可維護性的優先順序？
- 如何在架構層面保證這些屬性？
- 可接受的妥協是什麼？

## 創意空間

### 可選擇包含（根據專案需要）：
- **架構隱喻**：用現實世界類比解釋複雜架構
- **演進路線圖**：從 MVP 到最終願景的路徑
- **失敗場景分析**：系統可能如何失敗
- **架構決策樹**：不同情境下的架構選擇
- **團隊拓撲**：Conway's Law 的考量
- **成本模型**：架構對成本的影響

### 視覺化自由選擇：
- UML 圖表（類圖、序列圖、部署圖）
- 自定義架構圖
- 思維導圖
- 流程圖
- ASCII 藝術（如果合適）

## 最小必要約束

### 必須涵蓋（但形式自由）：
1. **系統組成**：主要組件及其關係（形式不限）
2. **技術選型**：主要技術決策及理由
3. **整合點**：與外部系統的介面

### 輸出要求：
- 位置：`memory-bank/designs/architecture/`
- 格式：Markdown（可嵌入各種圖表）
- 語言：根據團隊偏好（中文、英文或混合）

## 引導提示（非強制）

考慮以下問題，但不必全部回答：
- 如果用戶量增長 100 倍，架構需要改變嗎？
- 如果需要支持實時功能，當前設計是否適合？
- 新團隊成員需要多久才能理解這個架構？
- 這個架構是否過度設計了？
- 哪些部分可以購買而不是自建？

## 參考但不受限

可參考以下架構風格，但也可創新：
- Hexagonal Architecture
- Clean Architecture
- Event-Driven Architecture
- Microservices
- Serverless
- Monolith First
