# 需求理解引導 (/van)

## 核心目的
深入理解「問題」而非急於提出「解決方案」。最好的代碼是不需要寫的代碼。

## 探索框架（非固定格式）

### 1. 問題空間探索
**關鍵問題：**
- 現在的痛點是什麼？誰在承受這個痛苦？
- 如果不解決會怎樣？
- 這個問題值得用技術解決嗎？還是流程改進就夠了？

**表達自由：**
- 可用故事形式描述用戶的一天
- 可用數據呈現問題的嚴重性
- 可用對話形式展現痛點

### 2. 價值定義
**思考方向：**
- 成功看起來是什麼樣子？
- 如何測量成功？
- 誰會因此受益？如何受益？

**靈活表達：**
- OKR (Objectives and Key Results)
- User Journey Maps
- Value Stream Mapping
- 簡單的 Before/After 對比

### 3. 約束識別
**考慮面向：**
- 時間約束（截止日期、市場窗口）
- 資源約束（預算、團隊、技術）
- 品質約束（性能、安全、合規）

**記錄方式：**
- 可用 MoSCoW (Must/Should/Could/Won't)
- 可用風險矩陣
- 可用簡單列表

## 創意空間

### 可選擇性包含：
- **利益相關者地圖**：誰關心這個專案？他們的期望是什麼？
- **假設驗證計劃**：哪些假設需要先驗證？
- **競品分析**：其他人如何解決類似問題？
- **機會成本分析**：不做這個，我們可以做什麼？
- **用戶研究發現**：訪談、問卷、數據分析

### 形式完全自由：
- 傳統 PRD (Product Requirements Document)
- User Story Mapping
- Job-to-be-Done Framework
- Design Thinking Canvas
- 簡單的 FAQ 格式

## 最小約束

### 必須澄清（形式不限）：
1. **核心問題**：我們要解決什麼？
2. **目標用戶**：為誰解決？
3. **成功標準**：如何知道成功了？

### 輸出指引：
- 位置：`memory-bank/requirements/`
- 語言：使用團隊和利益相關者都懂的語言
- 長度：適可而止，寧缺毋濫

## 反思問題（可選）

- 這真的是用戶的問題，還是我們認為的問題？
- 有沒有更簡單的解決方案？
- 這個解決方案是否可持續？
- 我們是否在解決正確的問題？

**記住：好的需求文檔幫助團隊建立共識，而不是限制創意。**
